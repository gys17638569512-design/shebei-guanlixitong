# 数据开发结构与规范

## 1. 文档目标
- 统一后端数据结构、接口分层、命名规范、迁移规范与测试规范。
- 确保后续开发做到：结构清晰、职责清晰、变更可追踪、便于维护。
- 作为新版 PRD 的配套执行规范，所有新增模块优先遵守本规范。

## 2. 总体原则
- **优先增量扩展**：基于现有表与现有目录结构迭代，不做无必要的大规模推翻。
- **结构先于功能**：先定数据结构、权限边界、接口契约，再开始端口开发。
- **配置优先**：品类、型号、模板、品牌、状态项优先做成后台配置，不允许长期写死。
- **单一职责**：模型负责结构，服务负责规则，接口负责入参与出参，前端页面不直接承载业务规则。
- **可追溯**：关键写操作必须能追溯到操作者、来源端、变更前后快照。
- **可维护**：通用能力统一抽象，避免同一逻辑在多个端口重复实现。

## 3. 推荐后端结构

### 3.1 目录分层
```text
backend/
  api/
    routes/
  core/
  models/
  repositories/
  schemas/
  services/
  utils/
  tests/
  migrations/
```

### 3.2 分层职责
- `models`：数据库表结构、关系定义、枚举约束。
- `schemas`：请求体、响应体、列表查询参数、详情对象。
- `repositories`：数据库查询与持久化操作，避免把复杂查询散落在路由里。
- `services`：业务规则、状态流转、权限校验、跨表事务。
- `api/routes`：路由注册、参数接收、调用服务、返回统一响应。
- `core`：配置、数据库连接、认证、权限、审计、异常处理。
- `tests`：接口测试、服务测试、权限测试、流程测试。

### 3.3 禁止事项
- 禁止在路由函数中直接写大量业务逻辑。
- 禁止前端页面长期维护写死主数据。
- 禁止把核心可筛选字段全部塞进 JSON。
- 禁止跨端口各自维护一套同义状态值。

## 4. 数据表设计规范

### 4.1 命名规范
- 表名统一使用 `snake_case` 复数形式，如 `customer_accounts`。
- 字段名统一使用 `snake_case`，外键统一命名为 `{entity}_id`。
- 布尔字段统一以 `is_`、`has_`、`can_` 开头。
- 时间字段统一以 `_at` 结尾，如 `created_at`、`updated_at`、`bound_at`。
- 枚举字段统一使用可读字符串值，不使用魔法数字。

### 4.2 通用字段规范
建议所有核心主表都具备以下基础字段：
- `id`
- `created_at`
- `updated_at`
- `created_by_user_id` 或等价创建人字段
- `updated_by_user_id` 或等价更新人字段
- `is_active` 或 `status`

### 4.3 主键与外键规范
- 现阶段优先延续现有 `INT` 主键策略，减少兼容成本。
- 所有关联字段必须显式定义外键语义，即使某些历史表暂未真正加数据库级约束，也要在模型层保持一致。
- 关联删除策略要提前定义：是禁止删除、逻辑停用、还是允许解除关联，不能靠页面行为隐式处理。

### 4.4 状态与枚举规范
- 账号状态、模板状态、工单状态、维修状态分别定义，不混用。
- 同一业务对象的状态机只保留一份权威定义，前后端都从同一常量源读取。
- 状态变更要落审计，并记录来源端口。

### 4.5 JSON 字段使用原则
- 仅在扩展配置、动态参数、低频查询结构中使用 JSON。
- 核心筛选字段、排序字段、权限字段、关联字段不得只存 JSON。
- 需要版本锁定的模板主数据，应采用“主表 + 明细表”结构，不建议整包 JSON 覆盖。

### 4.6 版本化原则
- 模板、品牌配置、消息模板等需支持版本化的对象，必须区分“草稿、启用、归档”。
- 历史工单、历史报告必须锁定创建当时的模板版本。
- 禁止直接覆盖已被历史业务引用的关键模板内容。

## 5. 审计与追踪规范
- 关键写操作必须记录：
  - 操作者类型
  - 操作者 ID
  - 来源端口
  - 目标对象类型
  - 目标对象 ID
  - 变更动作
  - 修改前快照
  - 修改后快照
  - 请求链路 ID
- 员工账号、客户账号、品牌配置、模板配置、工单状态流转、签字确认必须进入审计。

## 6. 接口开发规范

### 6.1 接口分组
- 管理端接口按模块拆分，如：
  - `/api/v1/admin/users`
  - `/api/v1/admin/customers`
  - `/api/v1/admin/master-data`
- 客户端接口按客户场景拆分。
- 工人端接口按任务与执行场景拆分。

### 6.2 入参与出参规范
- 列表接口统一支持分页、关键字、状态、时间范围筛选。
- 创建与更新使用独立 Schema，不混用详情对象。
- 响应结构统一，至少包含：
  - `code`
  - `message`
  - `data`
- 分页响应统一包含：
  - `items`
  - `total`
  - `page`
  - `page_size`

### 6.3 权限规范
- 权限校验优先放在服务层或统一权限层，不放在前端兜底。
- 同一接口的角色范围要在代码与文档中同步声明。
- 微信端和 Web 端可以共用服务层，但要按端口限制能力范围。

## 7. 数据变更与迁移规范
- 所有表结构变更必须通过迁移脚本执行，不允许手工改库后再补代码。
- 迁移顺序建议为：
  1. 建表/加字段
  2. 回填默认数据
  3. 调整模型
  4. 调整服务与接口
  5. 补测试
- 涉及旧数据兼容时，优先采用“兼容期双字段/双写”策略，避免一次性切断旧逻辑。

## 8. 前端数据结构规范

### 8.1 推荐目录职责
```text
src/
  api/
  components/
  composables/
  constants/
  stores/
  types/
  utils/
  views/
```

### 8.2 前端开发原则
- `api` 只负责请求封装。
- `types` 统一维护接口类型与主数据类型。
- `constants` 只保留真正稳定的常量，不承载业务主数据。
- `stores` 统一管理登录态、权限、品牌配置、用户资料。
- `views` 不直接维护业务主数据枚举。

### 8.3 多端一致性原则
- 各端共享同一套品牌配置字段。
- 各端共享同一套状态文案与颜色语义。
- 各端通过统一主数据接口读取设备品类、型号、模板。

## 9. 开发实施顺序规范
- 第一步：确认表结构与字段语义。
- 第二步：确认权限矩阵与状态机。
- 第三步：输出接口清单与 Schema。
- 第四步：实现模型、仓储、服务、路由。
- 第五步：补充测试。
- 第六步：前端接入并移除硬编码。
- 第七步：联调与回归。

## 10. 维护约束
- 新增模块必须补文档，不允许只改代码不改说明。
- 新增主数据必须说明其来源、维护入口、使用端口、是否版本化。
- 删除字段或废弃接口必须给出兼容策略与清理计划。
- 所有总控审查优先检查：命名是否统一、职责是否越层、是否写死主数据、是否缺审计、是否缺测试。

## 11. 本项目当前执行结论
- 后续数据开发以“平台账号、客户账号、品牌配置、设备主数据、模板主数据、审计追踪”为核心底座。
- 后端开发统一按 `models / schemas / repositories / services / api` 分层推进。
- 前端开发统一按“配置读取优先、状态常量统一、页面不写死主数据”的方式推进。
- 总控智能体后续审查代码时，应以本规范作为结构性验收标准。
