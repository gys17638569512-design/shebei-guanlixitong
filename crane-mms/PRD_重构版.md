# 数字化起重机维修维保系统 PRD（重构版）

## 1. 文档信息
- **文档名称**：数字化起重机维修维保系统 PRD（重构版）
- **文档版本**：V2.0 Planning
- **适用对象**：产品负责人、开发负责人、总控智能体、端口子智能体、测试智能体、前端优化
- **文档目标**：作为下一阶段统一开发基线，替代旧版零散需求口径，统一产品范围、账号体系、端口矩阵、开发流程与验收标准

## 2. 产品重新定义

### 2.1 产品定位
本系统是一套面向起重机维保企业的**全生命周期设备服务管理平台**，服务于平台公司、客户公司、工程师三类主体，围绕设备建档、工单调度、现场执行、客户签字、报告归档、维修闭环、合规预警展开。

### 2.2 核心目标
- 用数字化系统替代纸质维保和维修流程
- 用账号体系和权限体系承载平台公司、客户公司、工程师的长期协作
- 用模板化和主数据化能力替代前端写死的设备品类、规格型号、检查内容、检修内容
- 用 Web + 微信双入口覆盖不同角色的完整场景和高频场景
- 用总控智能体 + 测试智能体建立稳定的多智能体开发闭环

### 2.3 核心价值
- **对平台公司**：统一账号、客户、设备、工单、品牌、报告与审计
- **对客户公司**：统一企业主账号、子账号、设备档案、历史报告、签字确认、报修入口
- **对工程师**：统一任务、打卡、拍照、检查、异常留痕、签字推动、离线同步
- **对交付与售后**：统一品牌展示、统一报告输出、统一审计追踪

## 3. 用户与组织模型

### 3.1 平台侧
- **平台管理员（ADMIN）**
  - 管理平台公司信息、品牌、员工账号、客户公司、客户主账号、模板、系统配置
- **业务/维保经理（MANAGER）**
  - 管理自己负责范围内的客户、设备、工单、维修进度、报告
- **工程师（TECH）**
  - 处理工单、执行检查、提交现场记录、发起维修闭环
- **平台员工本人**
  - 维护个人资料、密码、手机号、微信绑定、头像

### 3.2 客户侧
- **客户公司主账号（OWNER）**
  - 由平台创建，负责管理客户公司资料、子账号、设备查看、签字、报修
- **客户管理员（ADMIN）**
  - 受主账号授权，管理联系人、部分子账号、设备资料查看
- **客户签字账号（SIGNER）**
  - 负责签字确认、报告查看
- **客户查看账号（VIEWER）**
  - 负责设备和历史记录查看
- **客户报修账号（REPORTER）**
  - 负责快速报修和反馈

## 4. 产品矩阵（端口重构）

### 4.1 总体原则
- 每个业务端都必须同时支持 `Web` 和 `微信`
- **Web = 完整功能版**
- **微信 = 核心功能版**
- 不再把“一个业务端只对应一个载体”作为默认设计

### 4.2 端口划分
| 业务端 | Web 定位 | 微信定位 | 当前基础 |
| --- | --- | --- | --- |
| 管理端 | 完整后台管理 | 审批、提醒、看板、个人中心 | `frontend-pc` |
| 客户端 | 完整客户公司门户 | 签字、通知、报告、快速报修、个人中心 | `frontend-portal` |
| 工人操作端 | 完整任务与档案工作台 | 接单、打卡、拍照、检查、提交、离线同步 | `miniapp`（微信核心版） |

## 6. 账号体系需求

### 6.1 平台员工账号中心
- 管理员可新建员工账号
- 支持角色分配、直属关系、手机号维护、密码重置、账号启停
- 员工本人登录后可维护：
  - 姓名/昵称
  - 头像
  - 手机号
  - 微信绑定
  - 登录密码
- 所有改动必须进入审计日志

### 6.2 客户公司账号中心
- 平台在创建客户公司时，自动创建客户主账号
- 客户主账号登录后可创建子账号
- 子账号可区分角色：管理员、签字人、查看人、报修人
- 所有账号支持：
  - 密码管理
  - 手机号绑定
  - 微信绑定
  - 启用/禁用
  - 角色分配
- 联系人信息与登录账号分离，但允许关联

## 8. 设备主数据与检查检修模板需求

### 8.1 背景
当前设备品类、规格型号、部分检查项仍存在写死在前端的问题，后续必须全部改为后台可维护。

### 8.2 必须后台可配置的范围
- 设备品类
- 设备规格型号
- 默认技术参数
- 默认部件模板
- 检查模板
- 检修模板
- 维修模板
- 模板排序、启停、版本号

### 8.3 业务规则
- 创建设备时，从后台读取设备品类与规格型号
- 系统可按品类/型号自动带出默认部件模板
- 创建工单时，系统按品类/型号/工单类型锁定模板版本
- 工人端执行时直接读取后台发布模板
- 历史工单必须锁定当时模板版本，不受新模板影响

## 9. 权限体系

### 9.1 平台员工权限
- `ADMIN`：全局
- `MANAGER`：所辖客户与工单
- `TECH`：本人任务
- `SELF`：仅本人资料

### 9.2 客户公司权限
- `OWNER`：客户公司全量
- `ADMIN`：客户公司配置与部分账号
- `SIGNER`：在线签字
- `VIEWER`：查看设备与报告
- `REPORTER`：报修

### 9.3 权限原则
- 复杂配置优先放 Web
- 微信只开放高频核心功能
- 品牌、账号、模板、批量操作统一归 Web

## 10. 核心业务流程

### 10.1 员工账号流程
1. 管理员创建员工账号
2. 系统生成初始密码与强制改密标记
3. 员工首次登录后补全个人资料与绑定微信/手机号
4. 审计日志记录全过程

### 10.2 客户公司账号流程
1. 管理端创建客户公司
2. 系统自动生成客户主账号
3. 客户主账号首次登录并改密
4. 客户主账号创建子账号
5. 子账号按权限使用客户端 Web/微信

### 10.3 主数据配置流程
1. 管理员维护设备品类
2. 管理员维护规格型号
3. 管理员维护默认部件模板
4. 管理员维护检查/检修/维修模板
5. 工单创建时锁定模板版本
6. 工人端执行时读取锁定模板

### 10.4 工单执行流程
1. 管理端创建工单
2. 工人端接单
3. 工人微信端到场打卡
4. 工人填写检查/检修内容
5. 客户 Web/微信签字确认
6. 后端生成 PDF 报告
7. 客户端 Web 下载报告，管理端归档

### 10.5 维修流程
1. 工人或客户发起维修
2. 系统根据模板与型号加载维修项
3. 记录故障现象、维修内容、费用信息
4. 客户确认事实或费用
5. 归档维修报告

## 11. 技术与数据原则

### 11.1 总体原则
- 现有核心表优先增量扩展，不大规模重命名
- 新增客户账号表、品牌配置表、微信绑定表、模板主数据表
- 审计日志必须支持员工、客户账号、系统动作三类操作者
- 工单和维修单必须能追溯到具体发起人、签字人、确认人、来源端口

### 11.2 数据开发结构原则
- 数据结构必须清晰、分层明确、命名统一，方便持续维护与多人协作
- 后端推荐按 `models`、`schemas`、`repositories`、`services`、`api/routes` 分层开发
- 表设计统一遵循：主键明确、外键语义明确、状态枚举清晰、审计追踪完整
- 核心业务字段不得长期写入 JSON 或散落在前端硬编码中
- 模板、品牌、主数据类对象必须支持启停、排序、版本化

### 11.3 规范落地要求
- 后续数据库设计、接口设计、前端接入、测试回归，统一以 `数据开发结构与规范.md` 为执行标准
- 总控智能体在代码审查时，必须检查是否符合该规范中的分层、命名、迁移、审计与可维护性要求

### 12.2 智能体分工

#### 总控智能体
- 负责架构和任务拆分
- 负责接口契约、权限边界、跨端流程与集成验收
- 负责接收各子智能体提交并统一审查
- 负责将问题回派给对应子智能体

#### 管理端智能体
- 负责管理端 Web/微信
- 负责员工账号中心、客户管理、主数据配置、品牌配置、工单调度相关界面与接口

#### 客户端智能体
- 负责客户端 Web/微信
- 负责客户主子账号、设备查看、报告查看、签字、报修相关功能

#### 工人操作端智能体
- 负责工人端 Web/微信
- 负责任务、打卡、拍照、检查、提交、离线同步相关功能

#### 前端一致性专项智能体
- 负责统一颜色、字号、间距、卡片、按钮、表单、表格、空状态、权限提示
- 负责所有端口的品牌显示一致性

#### 测试智能体
- 负责接口、流程、状态机、权限、跨端闭环回归
- 负责按端口归类问题并提交总控

### 12.3 子智能体提交流程
1. 子智能体开发完毕
2. 子智能体自检
3. 提交总控智能体
4. 总控智能体初审
5. 测试智能体回归
6. 问题回派
7. 子智能体修复
8. 总控确认合并

### 12.4 子智能体提交标准
- 修改文件清单
- 功能说明
- 自测结果
- 影响范围
- 风险说明
- 待总控重点复核项

## 14. 验收标准

### 14.1 账号体系验收
- 平台员工账号可创建、可禁用、可改密、可自助维护
- 客户主账号可创建子账号并分配角色
- 微信与手机号绑定可追踪

### 14.2 主数据验收
- 设备品类与规格型号后台可维护
- 检查检修模板后台可维护
- 模板支持版本化
- 工人端不再写死检查内容

### 14.3 端口验收
- 管理端、客户端、工人端均具备 Web 与微信方案
- 微信只保留核心功能，Web 提供完整功能

### 14.4 流程验收
- 管理端建客户 → 客户建子账号 → 管理端派工 → 工人执行 → 客户签字 → PDF 下载，全链路打通
- 审计日志可定位到真实操作者
- 品牌与 Logo 在各端显示一致

## 15. 本版 PRD 的执行结论
- 后续开发必须以“账号体系 + 主数据体系 + 多端矩阵 + 多智能体协作”四条主线并行设计。
- 旧版需求中所有散落的口径，以本版 PRD 为准。
- 后续所有开发任务、接口设计、测试用例、智能体拆分，均需从本版 PRD 反向推导，不允许再各自理解。
