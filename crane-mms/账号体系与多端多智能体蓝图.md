# 账号体系、终端矩阵与多智能体协作蓝图

## 1. 文档目标
- 为下一阶段开发提供统一的账号体系、公司主体配置、多端产品矩阵与多智能体协作方案。
- 在不破坏现有 `users`、`customers`、`equipments`、`work_orders`、`repair_orders` 主链路的前提下，采用**增量式扩展**。
- 明确“Web 完整版 + 微信核心版”的双入口原则，并为总控智能体与测试智能体建立可执行的协作闭环。

## 3. 产品原则

### 3.1 端口原则
- 每个业务端都必须同时支持 `Web` 和 `微信`。
- `Web` 负责完整功能、配置项、复杂表单、批量管理、深度查询。
- `微信` 负责高频核心功能、消息触达、移动处理、现场操作。

### 3.2 账号原则
- 平台员工账号与客户公司账号**分层管理**，不混用一张表。
- 员工账号由平台管理员创建与管理，员工本人可维护个人资料。
- 客户公司由平台创建主账号，客户主账号再创建子账号。
- 联系人信息与系统登录账号**解耦**：联系人不一定有登录权限，登录账号也可反向绑定联系人。

### 3.3 数据原则
- 优先采用**加字段 / 加关联表**，避免大规模重命名旧表。
- 所有账号、主体信息、绑定关系、密码管理、品牌配置的写操作，都必须进入审计。
- 工单、维修单、签字、报修等行为必须能追溯到**具体操作者账号**。

### 4.3 现有表的扩展设计

#### 4.3.1 扩展 `users`
> 保留平台员工主表身份不变，增加“员工身份档案 + 自助资料中心”所需字段。

新增字段建议：
- `employee_code`：员工编号
- `display_name`：显示昵称
- `department`：部门
- `job_title`：岗位
- `email`：邮箱
- `avatar_url`：头像
- `is_active`：账号启停状态
- `must_change_password`：是否首次改密
- `password_updated_at`：最后改密时间
- `last_login_at`：最后登录时间
- `created_by_user_id`：创建人

#### 4.3.2 扩展 `customers`
> 继续保留“客户公司主表”定位，同时增加品牌资料和门户能力。

新增字段建议：
- `short_name`：公司简称
- `company_code`：客户编号
- `logo_url`：客户 Logo
- `portal_enabled`：是否开通客户端
- `owner_account_id`：客户主账号 ID
- `status`：`ACTIVE/INACTIVE`
- `industry`：行业
- `notes`：备注
- `created_by_user_id`：由哪位平台员工创建

#### 4.3.3 扩展 `contacts`
> 联系人继续服务于业务沟通，而不是强行承担登录功能。

新增字段建议：
- `department`：所属部门
- `is_primary`：是否主联系人
- `remark`：备注
- `account_linked`：是否已关联账号（布尔或由反向关系判断）

#### 4.3.4 扩展 `audit_logs`
> 当前 `audit_logs.user_id` 只能记录平台员工，无法承接客户主子账号、微信端行为，必须扩展为多身份审计。

新增字段建议：
- `actor_type`：`USER/CUSTOMER_ACCOUNT/SYSTEM`
- `actor_id`：操作者账号 ID
- `origin_app`：`ADMIN_WEB/ADMIN_WECHAT/CUSTOMER_WEB/CUSTOMER_WECHAT/WORKER_WEB/WORKER_WECHAT`
- `request_id`：请求链路 ID
- `old_value`：修改前快照

兼容建议：
- 旧字段 `user_id` 暂时保留，用于兼容现有代码。
- 新逻辑统一写 `actor_type + actor_id`，后续逐步淘汰只认 `user_id` 的调用方式。

#### 4.3.5 扩展 `work_orders`
> 为了支持客户主账号/子账号签字、不同端口提交，需要补足操作者追踪字段。

新增字段建议：
- `created_by_user_id`：平台创建人
- `reported_by_customer_account_id`：客户端发起人（如果从客户报修触发）
- `customer_sign_account_id`：实际签字确认账号
- `source_channel`：工单来源渠道
- `pushed_to_customer_at`：推送客户时间
- `completed_by_type`：`USER/CUSTOMER_ACCOUNT`
- `completed_by_id`：完工确认人

#### 4.3.6 扩展 `repair_orders`
> 同样需要记录是谁报修、谁确认、谁签字、从哪个端口进入。

新增字段建议：
- `customer_id`：直接归属客户公司
- `reported_by_customer_account_id`：报修发起人
- `confirmed_by_customer_account_id`：费用/事实确认人
- `source_channel`：报修来源渠道
- `created_by_type`：创建人类型
- `created_by_id`：创建人 ID

#### 4.3.7 关于现有 `check_templates` 的兼容处理
> 当前已有 `check_templates`，但能力还不够：仅按 `category` 管理，且 `items` 为 JSON，无法精细支持型号、检修内容、后台逐项维护。

兼容建议：
- 短期保留 `check_templates` 用于兼容现有接口。
- 中期以 `service_templates + service_template_items` 作为新标准模型。
- 逐步将 `check_templates` 迁移为只读兼容层或视图层。

## 4.4 后台可配置主数据范围
- 设备品类：后台可新增、修改、启停、排序。
- 设备规格型号：后台可按品类维护型号、规格、默认技术参数。
- 默认核心部件：后台可按型号维护“智能填充部件模板”。
- 检查内容模板：后台可按品类、型号、工单类型维护检查项。
- 检修/维修内容模板：后台可按维修场景维护检修项、故障项、建议项。
- 模板版本：所有模板修改后保留版本，历史工单锁定当时版本，不被新模板污染。

### 5.3 端口权限原则
- Web 端可承载复杂配置与组织管理。
- 微信端默认只开放“高频核心权限”，避免在移动端堆叠复杂配置页。
- 任何需要批量管理、品牌配置、复杂账号操作的功能，优先放 Web。

## 6. 核心业务流程蓝图

### 6.1 员工账号流程
1. 平台管理员创建员工账号。
2. 系统写入初始密码、强制改密标记、手机号、角色、直属关系。
3. 员工首次登录后完成改密、手机号确认、微信绑定、头像补全。
4. 所有动作进入审计日志。

### 6.2 客户公司账号流程
1. 平台在管理端创建客户公司。
2. 系统自动生成客户主账号，并将账号信息发送给客户。
3. 客户主账号首次登录后完成改密、微信绑定、联系人确认。
4. 客户主账号创建子账号并分配角色。
5. 子账号按权限查看设备、报告、签字或报修。

### 6.3 品牌配置流程
1. 平台管理员在品牌配置中心维护公司名称、系统名称、Logo、公章、页头页尾文案。
2. Web 端、微信端、PDF、短信/微信消息模板统一读取配置。
3. 客户公司品牌资料在客户维度独立维护，用于门户显示与报告抬头扩展。

### 6.4 跨端闭环流程
1. 管理端 Web 创建客户公司 + 主账号。
2. 客户端 Web/微信登录并维护子账号。
3. 管理端 Web 创建工单并派给工程师。
4. 工人微信端到场打卡、拍照、填写检查表。
5. 客户微信端或 Web 完成签字确认。
6. 后端生成 PDF，客户端 Web 下载，管理端 Web 归档。

### 6.5 主数据配置闭环流程
1. 管理端管理员维护设备品类与规格型号。
2. 管理端管理员维护对应的默认部件模板。
3. 管理端管理员维护检查模板、检修模板与维修模板。
4. 创建设备时，系统按品类/型号自动带出可选参数与默认部件。
5. 创建工单时，系统按品类/型号/工单类型锁定对应模板版本。
6. 工人端执行时直接读取后台模板，不允许前端写死默认检查项。

## 7. 多智能体协作蓝图

### 7.1 智能体角色与职责
| 智能体 | 主责范围 | 主要代码所有权 | 向谁提交 |
| --- | --- | --- | --- |
| 管理端智能体 | 管理端 Web + 管理端微信 | `frontend-pc`、管理后台相关 `backend/routers` 与 `services` | 总控智能体 |
| 客户端智能体 | 客户端 Web + 客户端微信 | `frontend-portal`、客户账号/门户相关后端 | 总控智能体 |
| 工人操作端智能体 | 工人端 Web + 工人端微信 | `miniapp`、未来 `worker-web`、工单执行相关后端 | 总控智能体 |
| 前端一致性专项智能体 | 统一设计规范与样式收口 | 各前端共享规范、主题变量、组件标准 | 总控智能体 |
| 测试智能体 | 接口、流程、回归测试 | `backend/tests`、E2E 脚本、联调验收脚本 | 总控智能体 |
| 总控智能体 | 架构、审查、集成与问题回派 | 全局设计与合并策略 | 最终对项目负责人负责 |

### 7.2 总控智能体职责
- 负责任务拆分与边界划分，避免多个智能体改同一职责模块。
- 审查接口契约、数据模型、权限边界、状态机与跨端流程。
- 检查是否符合“Web 全量、微信核心”的产品原则。
- 汇总测试智能体问题并回派给对应端口智能体。
- 负责最终集成验收与发布建议。

### 7.3 测试智能体职责
- 对每个业务端分别执行功能回归。
- 对跨端闭环进行端到端检查。
- 对账号、权限、品牌配置、多端显示一致性做专项验证。
- 将问题按“端口/后端/样式/权限/数据模型”分类后提交总控。

### 7.4 子智能体交付清单
每个子智能体提交时必须附带：
- 修改文件列表
- 功能说明
- 自测结果
- 影响范围
- 风险说明
- 需要总控重点复核的点

### 7.5 集成验收清单
- 数据模型是否兼容旧数据
- 平台员工账号是否支持自助资料维护
- 客户主账号与子账号是否权限分层
- Web 与微信是否功能定位正确
- 品牌名称与 Logo 是否全端一致
- 设备品类、规格型号、检查模板、检修模板是否全部可在后台维护
- 工单、维修、签字、报告链路是否闭环
- 审计日志是否能定位到真实操作者

## 9. 本蓝图的执行原则
- 先做数据底座，再做界面，不反过来。
- 先做角色与权限，再做复杂页面，不反过来。
- 先保证跨端流程闭环，再做装饰性扩展。
- 任何新端口都必须遵守统一品牌配置与设计规范。
