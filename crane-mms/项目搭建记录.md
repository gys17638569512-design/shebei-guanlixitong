## 遇到的问题及解决方案

### 1. 数据库连接问题
- **问题**：MySQL 连接失败
- **解决方案**：改用 SQLite 数据库，简化配置

### 2. 密码长度问题
- **问题**：bcrypt 密码长度限制
- **解决方案**：使用预生成的密码哈希值

### 3. 前端依赖问题
- **问题**：缺少 Pinia 和 Vue Router 依赖
- **解决方案**：安装相应依赖包

### 4. 目录结构问题
- **问题**：缺少前端视图目录
- **解决方案**：创建缺失的目录结构和视图文件

### 5. 前端依赖问题
- **问题**：缺少 Pinia 和 Vue Router 依赖
- **解决方案**：安装相应依赖包

### 6. Vite 缓存问题
- **问题**：Vite 无法识别新创建的文件
- **解决方案**：重启前端服务

### 7. 登录接口路由问题
- **问题**：登录接口返回 404 错误
- **解决方案**：统一配置路由前缀，确保登录接口路径正确

### 8. 数据库模型循环引用问题
- **问题**：WorkOrder 模型引用 Customer 模型时出现循环引用错误
- **解决方案**：调整模型导入顺序，确保所有模型都被正确注册到 Base 中

### 9. bcrypt 密码哈希问题
- **问题**：bcrypt 库出现版本读取错误，导致密码验证失败
- **解决方案**：暂时使用明文密码验证，绕过 bcrypt 库的问题

### 10. 数据库初始化问题
- **问题**：seed_data.py 脚本执行失败，无法插入用户数据
- **解决方案**：修改 seed_data.py 脚本，直接插入用户数据，绕过密码哈希步骤

## 遇到的问题及解决方案

### 11. 前端 Vite 别名解析失败白屏
- **问题**：`import ... from '@/...'` 导入报错找不到模块。
- **解决方案**：在 `vite.config.js` 中引入 `fileURLToPath` 手动配置 `resolve.alias` 指向 `src` 目录。

### 12. 老旧后端路由高耦合
- **问题**：早期的路由设计在接口层直接执行 `db.query` 带来长期的难以维护性。
- **解决方案**：一次性规范化建立 `services/` 与 `repositories/` 包结构，解耦执行逻辑。

## 需求补充与迭代规划（2026-03-18）

### 1. 管理端账号体系升级（以管理员为主）
- **当前现状**：
  - 本公司员工账号已具备基础的新增、编辑、删除能力，但仍以管理员单点维护为主。
  - 员工账号字段目前偏基础，缺少员工自助资料维护、微信绑定、手机号绑定、账号状态控制、密码策略等完整账号中心能力。
- **新增需求**：
  - 管理端管理员可以创建本公司员工账号，并维护角色、直属上级、手机号、初始密码、重置密码、启用/禁用状态等信息。
  - 员工本人登录后，也应拥有“我的账号中心”，可自行修改密码、维护昵称/姓名、绑定手机号、绑定微信、维护个人头像等资料。
  - 所有账号创建、重置密码、绑定关系变更、状态变更，均应纳入审计日志。
- **规划目标**：
  - 形成“管理员管组织账号，员工管个人资料”的双层账号管理模型。
  - 将员工账号从“系统登录账户”提升为“可持续运营的员工身份档案”。

### 2. 客户端账号体系升级（客户公司总账号 + 子账号）
- **当前现状**：
  - 客户侧目前以客户公司主联系人和登录手机号为主，缺少独立的客户公司账号树。
  - 现有客户门户登录更像“单联系人登录”，尚未形成客户公司内部多用户协作能力。
- **新增需求**：
  - 在管理端创建客户时，平台可以直接创建客户公司主体，并同步生成该客户公司的“总账号/主账号”。
  - 客户公司主账号登录后，应可在客户侧继续创建下属员工、负责人、设备管理员、采购或其他协同账号。
  - 客户公司账号需支持主账号与子账号关系、账号启停、手机号绑定、微信绑定、密码管理，以及不同成员的查看权限边界。
  - 客户公司主体信息由平台侧创建，但允许客户主账号在授权范围内维护联系人与下属账号。
- **规划目标**：
  - 将客户从“单联系人对象”升级为“企业主体 + 多成员账号体系”。
  - 形成平台管理员 → 客户公司主账号 → 客户子账号的三级管理结构。

### 3. 公司主体信息与 Logo 配置能力
- **新增需求范围**：
  - 本公司主体信息（系统归属公司）必须可维护，包括公司名称、简称、Logo、联系电话、联系地址、系统显示名称、对外展示文案、公章/品牌图片等。
  - 使用单位/客户公司主体信息也必须可维护，包括公司名称、Logo、地址、联系人、联系电话、展示简称等。
  - 本公司信息与客户公司信息都需要支持后续扩展到 PDF、登录页、侧边栏、门户首页、签章页、消息通知模板等界面。
- **规划目标**：
  - 消除当前系统内“名称和 Logo 硬编码”的问题。
  - 建立统一的品牌配置中心，让管理后台、客户端、报告文件、系统头部显示保持一致。

### 4. 前端一致性专项（单独子智能体负责）
- **专项原则**：
  - 前端界面美化不作为零散修补处理，单独设立“前端一致性与体验优化专项”。
  - 专门安排一个子智能体持续负责 `frontend-pc` 与 `frontend-portal` 的样式统一、组件规范、视觉层级、交互反馈、品牌展示一致性。
- **专项目标**：
  - 建立统一的颜色、间距、卡片、按钮、表格、表单、空状态、权限提示、页面头部等设计规范。
  - 梳理并统一管理端与客户端的登录页、导航、资料页、列表页、详情页、弹窗页、设置页的视觉语言。
  - 在后续接入公司信息和 Logo 配置后，保证不同端的品牌展示可联动且不失真。

### 7. 多端产品矩阵补充（管理端 / 客户端 / 工人操作端）
- **总体原则**：
  - 后续产品不再仅按“一个端对应一种载体”设计，而是按“一个业务端 = Web 完整版 + 微信核心版”双轨交付。
  - Web 端承载完整功能、复杂配置、批量操作、深度查询与管理能力。
  - 微信端保留高频核心功能，强调消息触达、快捷处理、移动场景和现场操作体验。
- **管理端**：
  - Web 端为主：完整提供账号管理、客户建档、设备管理、工单调度、品牌配置、报表审计、系统设置等后台能力。
  - 微信端为辅：保留消息提醒、待审批事项、关键看板、紧急工单查看、快捷确认与个人资料维护等核心能力。
- **客户端（客户公司端）**：
  - Web 端为主：完整提供客户公司资料、主子账号管理、设备档案浏览、历史记录查询、报告下载、故障报修、联系人维护等能力。
  - 微信端为辅：保留通知接收、报告查看、签字确认、设备状态查看、快速报修、个人资料维护等核心能力。
- **工人操作端（工程师端）**：
  - Web 端为主：完整提供任务列表、工单详情、照片记录、异常项填报、维修单处理、个人账号中心、历史任务归档等能力。
  - 微信端为辅：保留接单、导航、到场打卡、水印拍照、检查表填写、签字推动、完工提交、离线记录同步等现场高频能力。
- **规划目标**：
  - 三个业务端全部形成统一的“双入口架构”：每个端都支持 Web 和微信。
  - 保证“微信可办核心事，Web 可办全部事”，避免因终端限制导致流程断裂。

### 8. 多智能体协作开发机制补充
- **总体机制**：
  - 后续研发协作按“一个主要业务端对应一个主责智能体”的原则推进，并引入总控智能体与测试智能体做统一收口。
  - 所有子智能体完成阶段性工作后，必须先向总控智能体提交；由总控统一做代码合理性、依赖影响、跨端流程和接口一致性检查。
  - 总控发现问题后，按归属回传给对应子智能体修正，禁止多个子智能体在问题未闭环时直接交叉覆盖代码。
- **建议智能体分工**：
  - **管理端智能体**：负责 `frontend-pc` 与对应后台接口的管理端功能设计、页面实现、权限交互与配置中心。
  - **客户端智能体**：负责客户门户 Web/微信能力、客户主子账号体系、客户设备与报告视图、报修流程。
  - **工人操作端智能体**：负责工程师 Web/微信能力、现场作业流、检查表、打卡、水印、签字推动、离线同步。
  - **前端一致性专项智能体**：负责三个业务端的视觉规范、组件样式、品牌展示一致性与交互体验收口。
  - **测试智能体**：负责接口联调、跨端流程回归、状态机校验、权限边界验证、页面基础可用性检查。
  - **总控智能体**：负责架构把关、代码审查、接口契约校验、跨模块依赖检查、流程闭环验收、问题回派。
- **总控智能体职责**：
  - 统一维护任务拆分边界，避免多个子智能体改动同一职责区块。
  - 检查接口定义、权限模型、状态流转、品牌配置、账号体系是否在不同端保持一致。
  - 在每轮集成后组织一次整体验收，确认“接口能调通、流程能闭环、UI 不冲突、权限不串线”。
- **测试智能体职责**：
  - 对管理端、客户端、工人操作端分别执行针对性回归。
  - 对跨端场景执行完整验收，例如：管理员创建客户主账号 → 客户主账号创建子账号 → 工程师执行工单 → 客户微信签字 → Web 下载报告。
  - 将发现的问题按模块归类后提交给总控，再由总控派发至对应子智能体处理。
- **协作流程**：
  - 需求拆分 → 总控分派 → 子智能体开发 → 子智能体自检 → 提交总控 → 总控初审 → 测试智能体回归 → 问题回派 → 修复复检 → 总控确认合并。

- 当前专项将作为所有业务端智能体的上游设计约束，由总控智能体监督执行。
